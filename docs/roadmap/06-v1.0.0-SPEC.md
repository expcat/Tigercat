## 0. 版本概览

v1.0.0 是 Tigercat UI 组件库的**第一个正式发布版本**。这个版本标志着 Tigercat 从实验阶段进入稳定阶段，我们对库的 API、性能、质量做出正式承诺。

**版本目标**:

- API 冻结：承诺未来 12 个月内不会有破坏性变更
- 质量目标：所有指标都达到企业级标准
- 文档完整：面向生产环境的完整文档和最佳实践
- 社区就绪：完整的贡献者指南、社区治理结构
- 商业就绪：满足商业使用所有法律和技术要求

## A. 稳定性保证

### A.1 API 冻结策略

从 v1.0.0 开始，Tigercat 遵循 **Semantic Versioning** (SemVer) 规范：

| 版本范围       | 变更类型                     | 示例            | 破坏性 |
| -------------- | ---------------------------- | --------------- | ------ |
| v1.x.x         | Bug 修复、内部优化           | v1.0.0 → v1.0.1 | 否     |
| v1.y.0 (y > 0) | 新特性、新组件、向后兼容增强 | v1.0.0 → v1.1.0 | 否     |
| v2.0.0         | 破坏性变更、API 重设计       | v1.x.x → v2.0.0 | 是     |

### A.2 API 稳定性承诺

**在 v1.x 系列中，我们承诺**:

1. ✅ **组件 Props 不会移除或改变行为** — 已有的 Props 将继续支持
2. ✅ **Events 不会改变** — Events 名字、参数类型不变
3. ✅ **Slots 不会改变** — 已有的 Slots 名字不变
4. ✅ **无新增的必需 Props** — 所有新增 Props 都有合理的默认值
5. ✅ **Core 包 API 稳定** — `@expcat/tigercat-core` 的公开 API 不变
6. ✅ **CSS 变量名不变** — 主题 Token 的 CSS 变量名保持一致

**破坏性变更的处理**:

- 如需要 breaking changes，必须发布 v2.0.0
- Breaking changes 前至少 3 个月提前通知社区
- 提供完整的迁移指南和自动化工具

### A.3 版本计划

```
v1.0.0 (2026-03)  初始版本，API 冻结
  ↓
v1.1.0 (2026-09)  新增组件、新特性
  ↓
v1.2.0 (2026-12)  新增组件、新特性
  ↓
v1.3.0 (2027-06)  新增组件、新特性
  ↓
v1.x.x (持续维护)
  ↓
v2.0.0 (2028年后?)  如需要大幅改进，发布 v2
```

## C. 发布流程与检查清单

### C.1 发布前准备 (T-7 days)

- [ ] 冻结代码，不再接受新特性 PR（仅接受 bug 修复）
- [ ] 完成所有测试，确保 CI/CD 全绿
- [ ] 撰写详细的 CHANGELOG（所有 breaking changes、新特性、bug 修复）
- [ ] 撰写 Release Notes（市场宣传版本）
- [ ] 准备发布公告（GitHub, 社区博客、Twitter、微博等）
- [ ] 更新所有文档的版本号和发布日期

### C.2 发布前审查 (T-3 days)

- [ ] 代码审查：所有 breaking changes、新 API 都经过 senior review
- [ ] 文档审查：所有文档都经过编辑审查
- [ ] 国际化审查：所有语言翻译都经过审查
- [ ] 测试审查：测试覆盖率达到目标
- [ ] 性能审查：性能指标达到目标
- [ ] 安全审查：依赖更新无安全漏洞，代码无安全隐患

### C.3 发布日 (Release Day)

1. **npm 发布**

   ```bash
   # 确认当前分支是 main，没有本地改动
   git status

   # 更新版本号
   npm version 1.0.0

   # 构建
   npm run build

   # 发布到 npm
   npm publish
   ```

2. **GitHub Release 创建**
   - 创建 GitHub Release，标签为 `v1.0.0`
   - 附加 CHANGELOG.md 的内容
   - 上传构建的产物（如需要）

3. **文档站部署**
   - 部署文档站到生产环境
   - 验证所有链接都有效
   - 验证代码示例都可运行

4. **社区公告发布**
   - 发布 GitHub Release 公告
   - 发布社区博客文章
   - 发布 Twitter/微博 公告
   - 通知主要社区（GitHub, 掘金、V2EX 等）

### C.4 发布后 (T+7 days)

- [ ] 监控 GitHub Issues，快速响应用户反馈
- [ ] 准备 v1.0.1 patch（修复发现的 critical bugs）
- [ ] 社区反馈收集和分析
- [ ] 发布成功指标评估（下载量、Star 数等）

### D.1 长期维护承诺

| 版本 | 发布日期 | LTS 支持到 | 支持内容                         |

### D.2 Bug 修复 SLA

| Bug 等级 | 定义                           | 修复时间 | 发布时间      |
| -------- | ------------------------------ | -------- | ------------- |
| Critical | 无法使用、数据丢失、安全漏洞   | 24h      | T+3d (v1.0.x) |
| High     | 功能无法正常工作、严重性能问题 | 3d       | T+7d (v1.0.x) |
| Medium   | 边界情况、不符合预期           | 1 周     | 累积到 v1.0.x |
| Low      | 文档问题、细微交互             | 2 周+    | 下个版本      |

### D.3 安全政策

- 发现安全漏洞，请提交到 security@tigercat.dev (邮箱需要创建)
- 安全漏洞不公开，只在发布补丁后才披露细节
- 所有依赖都必须定期更新，确保没有已知安全漏洞
- 使用 `npm audit` 定期检查依赖安全性
